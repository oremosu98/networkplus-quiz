TB v3 Phase 4 dogfood smoke · 10 steps

Run these on http://localhost:3131. Verifies Simulate mode: panel open/close, device dropdowns, ping / ARP / DHCP animations, failure states, Validator Preview sequence, and Esc exit. If any step fails, ping back with which one + what you saw and I'll fix before we ship Phase 4.

01
Open Network Builder — confirm Simulate mode pill is now unlocked
In the left sidebar, click Network Builder. The canvas opens full-viewport. Look at the floating modebar centered at the top of the canvas. Confirm the Simulate pill is now clickable and not grayed out or showing a lock badge. In Phase 3 it was locked; Phase 4 ships it live. If Simulate still shows as locked, flag it — something in the mode-unlock logic did not land.
02
Enter Simulate with an empty canvas — panel opens, Validator Preview hidden, Send disabled
With a blank canvas (no devices dropped yet), click the Simulate pill in the modebar. The Simulate panel should open — a side panel or overlay showing protocol controls. Confirm: (a) the Validator Preview button is hidden (it only appears once a scenario is loaded), (b) the Send button is disabled (greyed out — nothing to simulate yet). If the panel does not open, or if Send is enabled on an empty canvas, flag it.
03
Drop a router + workstation, set IPs — dropdowns auto-fill, Send enables
Staying in Simulate mode, drop a Router from the palette onto the canvas, then a Workstation. Connect them with a cable (press c, click router, click workstation, press Esc). Auto-fill should give the router 192.168.10.1/24 and the workstation 192.168.10.10/24 with gateway 192.168.10.1. Now click each device in the Simulate panel. Confirm: (a) the source and destination dropdowns auto-fill with the devices and their IPs, (b) the Send button becomes enabled. If the dropdowns stay empty or Send stays disabled after you have two configured devices, flag it.
04
Send ping — round-trip motion completes, log shows “ping reached”
With the router and workstation in place and the Simulate panel open, select Ping as the protocol. Click Send. Watch the canvas: a packet should animate from the source device to the destination and back (round-trip). Confirm: (a) the motion completes fully without freezing, (b) the log panel fills with an entry containing something like ping reached (or equivalent success wording). If the packet animation stalls mid-flight or the log stays empty, flag it.
05
Send ARP — burst pulse + fan-out to subnet members + unicast reply, log fills
Switch the protocol selector to ARP. Click Send. ARP is a broadcast protocol: the animation should show a burst pulse radiating outward + a fan-out to all devices on the subnet, followed by a unicast reply from the target. The log panel should fill with both the broadcast line and the reply line — two distinct log entries. If the animation looks like a plain unicast ping instead of a broadcast burst, or if the log shows only one entry, flag it.
06
Send DHCP — four-stage DORA dance, one log line per stage
Switch the protocol selector to DHCP. Click Send. DHCP runs a four-stage exchange: Discover → Offer → Request → Acknowledge (DORA). The animation should step through all four stages on the canvas. The log panel should fill with four log lines, one per stage. Confirm all four stages appear in order. If only some stages animate or the log shows fewer than four entries, flag it with the count you saw.
07
Clear the workstation’s gateway → Send ping → failure shake + glow + red log row
Click the workstation on the canvas to select it. In the Inspector, find the Gateway field and clear it (select all, delete), then click elsewhere to blur. Now switch back to Ping in the Simulate panel and click Send. The packet should travel toward the workstation then stop. Confirm: (a) the workstation shows a red shake + glow animation indicating the failure, (b) the log panel adds a red log row reading something like WS-01 has no default gateway. (or equivalent). If the packet passes through silently or no failure animation fires, flag it.
08
Load the Star-topology scenario — Validator Preview button appears, click → all green, log shows N pairs reached
Click the Scenarios icon in the right-rail. In the picker, scroll to Topology types and select Star topology with central switch. Confirm the dialog and close the picker. Re-enter Simulate mode (click the Simulate pill). Now confirm: (a) the Validator Preview button is now visible in the Simulate panel (it was hidden in steps 2–3 because no scenario was loaded). Click Validator Preview. The sequence should run through all required device pairs automatically — each pair animates in turn. Confirm all pairs show green (reachable) and the log shows something like N pairs reached. If the button still does not appear, or any pair shows red on a valid star topology, flag it.
09
Break a workstation’s gateway in the scenario — Validator Preview shows failing pair red, others green, click red row re-plays that pair
With the Star topology loaded, click any workstation on the canvas and clear its Gateway field in the Inspector. Now click Validator Preview again. The sequence should run: the failing pair (involving the workstation with no gateway) shows red, while all other pairs remain green. Now click the red row in the log or results list. That pair should re-play its animation solo — just that pair, not the full sequence again. If all pairs show green after breaking the gateway, or clicking the red row does not re-play, flag it.
10
Press Esc mid-Validator-Preview — queue drains, panel closes, mode returns to Design
Re-open Simulate mode (with the Star topology still loaded and the broken gateway from step 9 still in place). Click Validator Preview to start the sequence. While the sequence is actively running (mid-animation, before it finishes), press Esc. Confirm: (a) the current animation drains gracefully (does not freeze or throw an error), (b) the Simulate panel closes, (c) the modebar switches back to Design mode. If pressing Esc mid-sequence causes a freeze, leaves the panel open, or does not return to Design mode, flag it.
