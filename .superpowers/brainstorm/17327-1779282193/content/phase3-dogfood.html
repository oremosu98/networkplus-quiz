Phase 3 dogfood smoke · 10 steps

Run these on http://localhost:3131. If any step fails, ping back with which one + what you saw and I'll fix before we ship Phase 3.

01
Open Network Builder from sidebar
In the left sidebar, click Network Builder (the FIRST entry). Page opens full-viewport — sidebar and topbar hide. Confirm the intent chip in the status bar reads Free Build. In the right-rail, verify the rrail icons are visible but the completion pill area is empty (no pill showing in Free Build). Any locked rrail icons should display their lock badges as expected.
02
Drop a router on the canvas
From the left palette, drag a Router onto the canvas and release. The device should auto-fill with IP 192.168.10.1/24 on interface 0. The Inspector panel should open automatically on the right side, showing the L3 interface block populated with: an IP field containing 192.168.10.1, a Mask field containing 255.255.255.0 (or /24), and a delete button for that interface. If the IP block is empty or the Inspector does not open, flag it.
03
Drop a workstation
From the left palette, drag a Workstation onto the canvas. Auto-fill should give it IP 192.168.10.10/24 and gateway 192.168.10.1. The Inspector should open for the workstation showing: IP field 192.168.10.10, Mask field 255.255.255.0 (or /24), and Gateway field 192.168.10.1. All three fields populated. If any field is empty, flag it.
04
Connect router → workstation via cable
Press c to enter cable mode (the cursor should change to crosshair and the status bar should show something like Cable mode · click two devices). Click the router, then click the workstation. A cable should appear connecting the two devices on the canvas. Press Esc to exit cable mode. Confirm the cable is visible. Note: you now have a 2-device Free Build canvas (router + workstation + 1 cable) — this is your Free Build backup that should persist through steps 5–9.
05
Open the scenarios picker — pick “Star topology with central switch”
Click the Scenarios icon in the right-rail. The picker panel slides out from the right. Scroll to the Topology types category and click Star topology with central switch. A confirm dialog appears (your Free Build has content — it will be saved). Click Accept. The canvas loads with the pre-configured star topology: 1 switch + 1 server + 3 workstations + 4 cables, all with their IPs already configured. Close the picker with the × button. Confirm: (a) the intent chip now reads Lab · Star topology with central switch and (b) the completion pill in the right-rail reads GOALS MET with an accent-filled (bright) style. If the pill says anything else, check that all required devices are on canvas with valid IPs.
06
Click the completion pill — diagnostic drawer opens
With the picker closed, click the GOALS MET pill in the right-rail. A diagnostic drawer should slide in from the right. In the empty (all-passing) state, verify the drawer shows: an eyebrow reading DIAGNOSTICS / Goals met (or similar), a body line reading All paths reachable or equivalent, a sub-line like Every required cable type-pair has a working route, and a footer reading UPDATED LIVE AS YOU EDIT (or similar). No failure rows should be visible. If the drawer does not open or shows failure rows, flag it.
07
Close the diagnostic drawer
Click the × button inside the diagnostic drawer (or click outside it if that closes it). The drawer should slide out and the right-rail should return to its normal state with the completion pill visible again. Confirm the pill still reads GOALS MET after closing.
08
Break the scenario — verify diagnostic updates
Click WS-01 (or any workstation) on the canvas to select it. The Inspector opens. Find the Gateway field and clear it (select all and delete), then click elsewhere to blur. The completion pill should flip from GOALS MET to 1 TO GO (dimmed style). Now click the pill to open the diagnostic drawer. Verify the drawer shows one failure row with a title like SRV-01 → WS-01 (or a similar unreachable-pair label) and a reason like WS-01 has no default gateway. If the pill does not flip or the drawer shows no failure row, flag it.
09
Fix the failure — verify recovery
While the diagnostic drawer is open showing the failure row, click the failure row. The drawer should close and the Inspector should open for WS-01 (auto-selected). Re-type the gateway 192.168.10.1 in the Gateway field and blur. Now click the pill again to re-open the drawer. The drawer should show the empty (passing) state again — All paths reachable and no failure rows. The pill in the right-rail should have flipped back to GOALS MET (accent-filled). If the pill does not recover or the drawer still shows failures, flag it.
10
Reload mid-Lab — state persists + Free Build restores
With the scenario still loaded (Lab mode, GOALS MET), reload the page. Navigate back to Network Builder from the sidebar. The scenario should still be loaded — same devices with their IPs, intent chip reading Lab · Star topology with central switch, completion pill showing GOALS MET. Now click the intent chip. The canvas should clear and the intent chip should change to Free Build. Your original 2-device Free Build canvas from steps 2–4 (router + workstation + 1 cable, with their auto-filled IPs) should restore. Critical: both Lab state and the Free Build backup must survive the reload. If the Free Build devices are missing after clicking the intent chip, flag the backup-restore flow.
When all 10 pass: tell me “ship Phase 3” (or similar) and I’ll merge feat/tb-v3-phase3 into main, bump the version, expand the CLAUDE.md row, push to main, monitor CI, and prod-verify.

If any step is broken or feels off: tell me which step + what you saw + what you expected, and I’ll fix on the branch before we ship.

Phase 3 known limitation: Scenarios with L3 transit paths (spine-leaf two-spine-to-spine routing, distant-router reachability across multiple hops) currently use a smart-pick strategy that succeeds if any representative pair can route — not if every possible pair routes. A future Phase 3b can tighten this by adding static-route configuration to the scenario definitions so the validator can verify full-mesh reachability explicitly.
