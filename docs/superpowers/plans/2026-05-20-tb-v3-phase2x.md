## Authoring template (locked)

Each scenario MUST follow the v5.8.0 mesh-topology shape verbatim:
- `id` — kebab-case slug, unique
- `title` — sentence-case display name
- `category` — one of the 7 spec values
- `objectiveRefs` — N10-009 objective number strings (e.g. `['1.2']`)
- `startingState`:
  - `devices[]` — `{id: 'sc_<slug>_<n>', type, x, y, label}` (x ∈ [40, 1760], y ∈ [40, 1060], all multiples of 40)
  - `cables[]` — `{id: 'sc_<slug>_c<n>', fromId, toId, type}` (type defaults to `'cat6'`, use `'fiber'` for long-haul/DC)
  - `viewport: { x: 0, y: 0, zoom: 1 }`
- `brief` — 1 paragraph (~2-3 sentences) explaining the topology + its trade-off
- `examRelevance` — 5 keys (`overview`/`howItRoutes`/`keyDevices`/`keyConcepts`/`examRelevance`), 1 sentence each, concise
- `completion`:
  - `requiredDevices: [type, ...]` — unique device types
  - `expectedCount: { type: n, ... }` — match starting-state counts
  - `requiredCables: [{from: type, to: type}, ...]` — pair-match cables (order-agnostic)

**Device-type vocabulary** (from existing Phase 1 palette + v5.8.0 scenarios): `router`, `switch`, `l3-switch`, `firewall`, `vpn`, `cloud`, `internet`, `wlc`, `ap`, `smartphone`, `laptop`, `server`, `workstation`. Use ONLY these.

## 17-scenario manifest

### Topology (+2)

**5.x.1 `ring-topology`** — obj 1.2 — 5 routers in closed loop · 5 cables forming ring · failure-domain on link break · contrast with star/mesh

**5.x.2 `point-to-point-topology`** — obj 1.2 — 2 routers directly connected · 1 cable · simplest WAN building block · contrast with multi-access

### Architecture (+3)

**5.x.3 `spine-leaf-fabric`** — obj 1.6 — 2 spine + 4 leaf L3-switches · 8 fiber cables (every leaf ↔ every spine) · modern data-center fabric · contrast with 3-tier

**5.x.4 `collapsed-core`** — obj 1.6 — 2 L3-switches (collapsed core+distribution) + 4 L2 access · 6 cables · SMB/branch compromise · contrast with 3-tier

**5.x.5 `soho-network`** — obj 1.6 — 1 router (gateway) + 1 switch + 1 AP + 2 workstations + 1 laptop + 1 smartphone · ~6 cables (laptop+phone wireless to AP) · home/small-office baseline

### WAN (+3)

**5.x.6 `mpls-wan`** — obj 1.6 — 1 cloud (MPLS provider) + 1 HQ router + 4 branch routers · 5 fiber cables (each router → MPLS cloud) · provider any-to-any · contrast with hub-spoke

**5.x.7 `partial-mesh-wan`** — obj 1.6 — 4 routers, partial mesh (4 of 6 possible links) · 4 cables · cost/reliability trade-off · contrast with full mesh

**5.x.8 `dual-isp-failover`** — obj 1.6 — 1 router + 2 internet nodes (primary/secondary ISP) + 1 firewall + 1 switch · 5 cables · redundant internet · BGP/failover concepts

### Cloud (+3)

**5.x.9 `public-cloud-only`** — obj 1.8 — 1 internet + 1 firewall + 2 servers + 1 workstation (cloud-native, no on-prem) · 5 cables · IaaS deployment · contrast with hybrid

**5.x.10 `multi-cloud`** — obj 1.8 — 2 clouds + 1 router (VPN bridge between them) · 2 fiber cables · workload portability + vendor lock-in trade-off

**5.x.11 `direct-connect-private-link`** — obj 1.8 — 1 on-prem switch + 1 router + 1 cloud · 2 fiber cables (dedicated private link, no internet) · low-latency cloud · contrast with VPN-over-internet

### Wireless (+2)

**5.x.12 `wireless-mesh`** — obj 2.4 — 1 router + 1 switch + 3 APs (mesh backhaul) + 1 smartphone client · 5 cables (router-switch, switch-AP-root, AP1-AP2, AP2-AP3 modelled as wireless bridges drawn as cables) · self-healing wireless

**5.x.13 `wireless-bridge-p2p`** — obj 2.4 — 2 switches + 2 APs (configured as wireless bridge) + 1 laptop · 4 cables (switch1-AP1, AP1-AP2 wireless bridge, AP2-switch2, switch1-laptop) · building-to-building link

### Security (+3)

**5.x.14 `zero-trust-segmented`** — obj 4.1 — 1 firewall + 3 servers + 3 workstations (each microsegmented through the FW) · 6 cables (every endpoint to FW) · "never trust, always verify" · contrast with implicit trust LAN

**5.x.15 `bastion-jump-host`** — obj 4.1 — 1 internet + 1 firewall + 1 bastion server (jump host) + 1 switch + 2 internal servers · 5 cables · controlled inbound admin access · contrast with VPN

**5.x.16 `nac-802-1x`** — obj 4.1 — 1 NAC/AAA server + 1 switch + 1 router + 3 workstations · 6 cables · port-based authentication · 802.1X + RADIUS

### VLAN (+1)

**5.x.17 `l3-switch-svi`** — obj 2.1, 2.3 — 1 L3-switch + 4 workstations across 2 VLANs · 4 cables · SVI-based inter-VLAN routing (no router needed) · contrast with router-on-a-stick
