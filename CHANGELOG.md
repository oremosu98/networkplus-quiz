| v4.43.0 | **Keyword highlighting in question stems** — new product feature (minor version bump, not maintenance). CompTIA exam stems use specific trap words that change meaning: *"Which of the following is NOT..."*, *"MOST secure option EXCEPT for..."*, *"The FIRST step in the troubleshooting methodology..."*. Missing these words is the #2 wrong-answer mode after "I didn't know it" — test-takers skim and miss the negation or superlative. This release auto-highlights them at render time to train the eye to catch them on the real exam. **14-keyword dictionary**: `NOT`, `EXCEPT`, `CANNOT` (negation), `MOST`, `LEAST`, `BEST`, `WORST`, `PRIMARY` (superlatives), `FIRST`, `LAST`, `NEXT` (sequence), `ALWAYS`, `NEVER`, `ONLY` (absolutes). Deliberately **excludes** `all`, `none`, `any`, `could`, `should`, `would`, `which` — those are too common in plain prose, would produce visual noise. UAT regression-guards the exclusion list. **Rendering approach**: new `setQuestionText(el, raw)` helper that does `el.innerHTML = highlightExamKeywords(escHtml(raw))` — single entry point for all 5 render paths (quiz stem, exam stem, review page, CLI sim scenario, topology scenario). Escapes HTML first (XSS-safe since `q.question` is AI-generated), then wraps keywords in `<strong class="exam-keyword">`. Word-boundary regex with `gi` flags so case is preserved (lowercase "not" in prose stays lowercase but still styled — doesn't visually conflict since the styling is background + color, not text-transform). **CSS**: subtle chip styling — bold + accent-light color + 10% accent background + inset bottom-border underline. Light theme gets a slightly darker accent variant for contrast. NOT a big shout — keywords should catch the eye without dominating the sentence. **What's NOT touched**: Explain Further content, Topic Deep Dive, Topic Brief, Analytics pages — those are teacher-generated content, not questions. Keywords only appear in question stems where the pedagogy win lives. **UAT**: 44 new v4.43.0 assertions — constants/helpers present, all 14 keywords covered, 7 ambiguous words explicitly excluded, 5 render paths wired up, 2 regression guards against old `textContent`-based rendering, 3 CSS presence checks, 2 XSS-safety guards (escape before highlight, null-input handling), and a 7-case behavioral smoke test that verifies upper-case / mixed-case / multiple-keywords / plain-prose / word-boundary / excluded-word / escaped-HTML-entity behavior all work correctly. | v4.42.4 | **Readiness algorithm — within-domain question-count weighting fix** — user asked for a correctness audit of the exam readiness calculator. Audit surfaced one real flaw and three deferrable tuning preferences. Shipped the correctness fix; filed the rest on Board #1 gated on SaaS-pivot user data. **The flaw**: `getReadinessScore()` was computing domain accuracy as the simple average of per-topic percentages (`pctSum / count`), giving equal weight to a topic with 2 questions and one with 200. The real CompTIA exam weights every question equally — so a user who is 95% on OSPF (20 weighted Q's) and 50% on BGP (4 weighted Q's) has a real-exam-equivalent implementation accuracy of `(19+2)/(20+4) = 87.5%`, not the simple average of 72.5%. The v4.42.3 catalog expansion (40 → 50 topics) made the flaw more visible because new child topics start with 0 questions while parent umbrellas have many — high topic-count variance makes the averaging distortion bigger. **The fix**: ~10 lines in `getReadinessScore()`. Swap `pctSum += pct; count++` aggregation for `wCorrect += topicMap[t].wCorrect; wTotal += topicMap[t].wTotal` — then domain average is `wCorrect / wTotal * 100` across all topics in the domain, properly weighted by question count. Unstudied-domain behavior preserved (still computes 0 and drags the score down). The 40/25/20/15 signal weights (accuracy / coverage / recency / volume), 420-870 scaling, domain weights (23/20/19/14/24), difficulty multipliers (Hard 2.0 / Exam 1.5 / Found 1.0), exam-mode boost 1.3×, 7-day recency boost 2×, 14-day recency decay window, and 500-Q volume cap are all **untouched** — those were reviewed and deemed correct. **Deferred tuning opportunities** (filed as issue #139, gated on SaaS-pivot user data): (1) Recency denominator is `totalTopics` not `studiedTopics`, which subtly makes recency signal partially a coverage signal, redundant with the Coverage signal itself; (2) No PBQ weighting in accuracy calc — real CompTIA exam weights PBQs heavier than MCQs, but the algorithm treats them the same; (3) Triple-penalty for low coverage (coverage × 0.5 under 5 topics, unstudied domains = 0 accuracy, small recency numerator) — mathematically fair but potentially discouraging for early users. Each of these is a judgment call without clear "right" answer for a single-user tool; revisit when SaaS has real user data to observe what correlates with pass rates. **UAT**: 10 new v4.42.4 assertions — 5 structural regression guards (new `wCorrect/wTotal` aggregation present, old `pctSum/count` pattern gone, `bucket.wTotal > 0` division-by-zero guard) + 5 behavioral smoke assertions on a synthetic fixture that proves (a) new logic gives 87.5% on unequal-Q-count data, (b) old logic would have given 72.5%, (c) new vs old differ by >10% when Q counts vary, (d) new and old agree when Q counts are equal (shows the fix is a strict upgrade, not a swap), (e) unstudied domain still returns 0.
