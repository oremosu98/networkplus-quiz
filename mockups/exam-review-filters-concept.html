NACLs at the subnet level + Security Groups at the instance level are the primary mechanisms for stateless and stateful filtering within a VPC. VPC Peering connects separate VPCs (not isolation within one). VPC Flow Logs are for monitoring, not control.

Design notes
  • 5 chips: All / ✓ Correct / ✗ Incorrect / ⚑ Flagged / ↷ Skipped. Counts in pills next to each label so the user can see at a glance how many of each type they have.
  • Tier-colored active state: Correct chip turns green when active, Incorrect turns red, Flagged turns yellow, Skipped turns gray. Visual reinforcement of what's currently filtered.
  • Disabled when count is 0: Skipped chip in the screenshots above shows the disabled state (skipped count = 0 since you answered all questions). User can see it exists but can't click it. Same applies if Flagged = 0.
  • Sticky header: Filter row sticks to the top of the scroll area as you scroll through questions. Backdrop-blur keeps it readable over content below. Disappears at <540px to save vertical space.
  • "Showing N of M" indicator: Right-aligned counter so the user always knows how many questions match the current filter.
  • Per-question tags: each question card now shows its status tag (✗ Incorrect, ⚑ Flagged, etc.) + domain in a meta row above the stem. Useful even when filtering — confirms which filter matched this card.
  • Q numbering preserved: cards still show "Q 14 of 90" so the user can see which actual exam question they're reviewing — order is preserved within the filter.
  • Empty state: when a filter returns 0 results (e.g., user scored 100% so Incorrect = 0), show a celebratory empty state with a "Show all" reset button.
  • Multi-select filters?: not in v1. Keep it simple — single-select chips with ergonomic counts. If you later want "Incorrect AND Flagged" the chips can become checkbox-style.
  • Implementation cost estimate: ~80 LOC in app.js (count computation + filter state + re-render on chip click), ~60 LOC in styles.css, ~30 LOC in index.html (filter-row HTML). No new state pipeline — just a render filter applied to the existing review questions array.
