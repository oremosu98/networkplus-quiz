🔐 Phase C — Cert-app Supabase integration

Bring the Supabase auth client into the cert app (networkplus.certanvil.com) so a user signed in on apex (certanvil.com) is recognised on subdomain via the apex cookie. Pure identity layer — does NOT include progress sync (Phase D), API key proxy (Phase E), migration (F), or paywall (G). Foundation for all of those, but visible-on-its-own as: account pill in topbar, sign-in nudge moments, and shared sign-out across surfaces. Anonymous experience preserved — sign-in is optional, never gating the product.

Architectural decisions (handshake before code)

1. Sign-in UI stays centralised on landing. The cert app does NOT host the auth modal. Click "Sign in" on cert app → redirects to certanvil.com/?action=signin&return=https://networkplus.certanvil.com → modal opens on landing → after success, browser bounces back to cert app. One source of auth UI = no version drift; landing's modal is already battle-tested.

2. Apex cookie is the session-sharing mechanism. Already configured in landing/lib/supabase.js via cookieOptions: { domain: '.certanvil.com' }. Cert app loads the same lib + Supabase client → sees the same cookie → recognises the same session. No token-passing dance. Sign out on apex → cookie cleared → cert app signed out on next page load.

3. Anonymous experience preserved entirely. Users CAN use the cert app without ever signing in (matches today's behaviour). Sign-in is offered but never required. Localstorage (nplus_*) keeps working. Phase F (anon → authed migration) is what actually moves the data over later — Phase C just adds the identity awareness.

4. Sign-in nudge appears at high-value moments only. Three nudge surfaces: (a) subtle persistent topbar "Sign in" pill, (b) one-time soft banner after first quiz completion ("Your progress is local-only — sign in to sync across devices"), (c) one-time soft banner after diagnostic completion ("Want this Pass Plan saved across devices?"). All dismissible. Never on every page load. Never a hard block.

5. Account pill copy + dropdown match landing exactly. Same component visual language across surfaces. When signed in: avatar (initial in accent gradient) + truncated email + chevron. Click → dropdown with email + tier badge + 3-4 actions + sign-out. The cert app's existing avatar circle in the topbar gets REPLACED by this richer pill when authed, falls back to "Sign in" pill when anonymous.

6. Tier display reads from cert_entitlements. Postgres trigger from Phase A schema auto-grants a free Network+ entitlement on signup. Dropdown shows "Free tier · Network+ unlocked" or similar. Future Pro tier upgrade (post-Phase G) flips this to "Pro · all certs unlocked".

7. Sign-out clears everything cleanly. Click Sign out in dropdown → Supabase signOut() clears the apex cookie + localStorage session entry → cert app re-renders with anonymous topbar → small green "Signed out" toast for confirmation. All other tabs at certanvil.com + *.certanvil.com pick up the change via onAuthStateChange → simultaneous sign-out across the whole product.

8. What this DOES NOT do (and why): No progress sync (Phase D — separate ship; localStorage stays the source of truth until then). No API key proxy (Phase E — users keep BYOK until then). No paywall (Phase G — Pro features still open until Stripe wires up). Phase C is identity ONLY. The cert app knows WHO you are; it doesn't yet do anything different based on that knowledge. That comes in D/E/F/G.

Implementation plan · estimates · file map
Files to create / touch (cert app side)

app/lib/supabase.js (new, copy of landing/lib/supabase.js) — same Supabase client init, same apex cookie config. Could also be loaded directly from a shared CDN-style URL but a per-app copy is simpler.

app/auth-state.js (new, ~150 LOC) — slim auth-state wrapper for the cert app. NOT the full modal (that lives on landing). Listens to onAuthStateChange, renders the topbar account-pill / sign-in-pill / dropdown, handles sign-out, dispatches events for greater-app code that needs to react. Anonymous-progress detection logic NOT here (that's Phase F migration territory).

index.html (cert app) — add <script> tags for Supabase CDN + lib/supabase.js + auth-state.js. Add new container DOM for account pill / dropdown in topbar (hidden by default, populated by auth-state.js).

app.js (cert app) — replace existing avatar-circle render with the new account-pill / sign-in-pill toggle. Hook into renderAppSidebar or wherever topbar is built today. Add the optional sync-hint banner render (post-first-quiz / post-diagnostic surfaces).

styles.css (cert app) — copy account-pill + dropdown CSS from landing/styles.css. Add sync-hint banner CSS. Add synced-badge CSS.

Files to touch (landing side)

landing/auth.js — handle the new ?return= URL param. After successful auth, redirect to return if it's a trusted *.certanvil.com origin (allow-list to prevent open redirect).

landing/index.html + landing/pricing.html — sign-in modal already auto-detects URL params; just need ?action=signin support added (matches existing ?action=diagnostic pattern in cert app). Auto-opens modal on page load if param present.

Time estimate

~2-3 hours total:
• Cert-app auth-state.js + topbar wireup: ~90 min
• Landing return-URL handler + auto-open-modal logic: ~30 min
• Sync-hint banner surfaces (post-quiz, post-diagnostic): ~30 min
• CSS polish + cross-tab sync verification: ~30 min

Acceptance criteria (when this ship is "done")

(a) User signed in on apex sees account pill on cert-app subdomain on next page load. (b) Sign-out from cert-app subdomain clears session everywhere within 1 page navigation in any open tab. (c) Anonymous users still get full app functionality — no regressions on the localStorage-based experience. (d) The "Sign in" pill is the only auth affordance in cert-app topbar (no full modal duplicated). (e) Click "Sign in" from cert-app → lands at certanvil.com/?action=signin&return=... → modal opens → after auth, returns to cert-app subdomain with session active. (f) Sync-hint banner surfaces ONCE per dismissal at the documented trigger points; never spammy.

Phases this unblocks

Phase D (progress sync): Now that the cert app knows the user identity, it can mirror-write quiz_history rows to Supabase per session.
Phase E (server-side API key proxy): Cert app reads tier entitlement from Supabase, gates AI calls, server-side endpoint accepts authenticated requests.
Phase F (anon → authed migration): When sign-in fires, cert app reads localStorage nplus_* keys + offers one-click "Import to your account?" prompt. Schema's already in place.
Phase G (Stripe paywall): Account dropdown gets an "Upgrade to Pro" CTA; paywall logic gates premium features.
