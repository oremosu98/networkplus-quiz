☁️ Phase C + D + F bundled — Cloud sync end-to-end

Combined ship: identity (Phase C) + progress sync (Phase D) + anon→authed migration (Phase F). When a user signs in, their progress actually moves to the cloud and syncs across devices — matching the user mental model. Avoids the awkward "Phase C only" middle state where users sign in and notice nothing changed for their data. 7 states stacked top-to-bottom showing the full anonymous → first-signup → daily-use → cross-device → sign-out flow.

Why bundle these three phases (handshake)

1. Phase C alone is misleading. If users sign in but their progress stays local-only, the natural mental model breaks ("I have an account — surely my data's saved?"). Shipping the identity layer without the data layer creates a UX bug where the value proposition is implied but not delivered.

2. C + D + F share the same auth-state-change pipeline. All three react to the same SIGNED_IN event from onAuthStateChange. Phase F (one-time migration) fires first; Phase D (ongoing sync) takes over once migration completes. Wiring them separately means doing the same plumbing three times.

3. The honest "you're now synced" UX requires all three. Sync status indicator (Phase D), migration banner (Phase F), account pill + dropdown showing sync state (Phase C) — each is incomplete without the others.

Architectural decisions (locked-in)

4. Local-first, cloud-mirrored. localStorage stays the read source-of-truth (zero latency, works offline). Writes go to localStorage immediately AND queue for cloud sync. Sync worker batches Supabase writes every ~30s or on pagehide. No write-blocking on slow networks.

5. Migration is one-time + opt-in. First sign-in fires migration check. If localStorage has nplus_* data + Supabase has nothing for this user → show prompt. Never auto-merge per the v4.81 data-safety discipline. profiles.metadata.migration_v1_at records the migration; never re-prompts.

6. Sync conflict resolution. quiz_history = append-only, no conflict possible (each row is a unique event). Settings (exam_date, daily_goal) = last-write-wins by timestamp. Streak = max-value (you can't have a "smaller" streak; if cloud says 13 days and local says 5, cloud wins). First-time hydrate on new device = cloud-wins entirely.

7. Sync status UI lives in dropdown, not topbar. Topbar account pill stays clean (avatar + email + chevron + small status dot). Dropdown reveals the full status: "● Synced 5s ago [Sync now]". Avoids constant "syncing" noise in the user's peripheral vision.

8. Sign-out preserves localStorage. Anonymous mode after sign-out continues to work — all the user's local data is intact. Their cloud data also stays untouched in Supabase (separate storage; sign-out doesn't delete account). Sign back in → reconnects.

9. Offline mode keeps working. Sync queue retries on next online event. Subtle yellow indicator in dropdown if sync is pending: "● 3 changes will sync when online".

10. What's NOT in this bundle (intentional): Server-side API key proxy (Phase E — users still BYOK after this ship). Stripe paywall (Phase G — Pro features still ungated). Both deserve their own dedicated sessions. This bundle is "your data is saved and syncs" — not "you're paying for things."

Implementation plan · file map · time
Phase C — Identity layer (~2 hr)

app/lib/supabase.js (new) — Supabase client init copy from landing.
app/auth-state.js (new) — slim auth state listener, account-pill render, dropdown render, sign-out handler.
index.html — script tags + topbar account-pill container.
app.js — replace existing avatar circle with account-pill / sign-in-pill toggle. Sync-hint banner render.
styles.css — port the account-pill + dropdown CSS from landing (already proven in landing's auth.js).

Phase D — Sync layer (~3 hr)

app/sync.js (new, ~300 LOC) — sync worker with queue, batch, retry. Hooks into saveToHistory(), saveStreak(), drill mastery setters, etc. Handles conflict resolution per the locked-in policy. Queues offline events for retry.
Hydration logic — on SIGNED_IN if no local data + cloud has data → fetch + merge.
supabase/schema.sql — add tables/columns for streak (jsonb on profiles), drill mastery (per-cert jsonb).
Sync status events emitted via window event bus → consumed by sync-dot + dropdown UI.

Phase F — Migration layer (~1 hr)

app/migration.js (new, ~150 LOC) — first-sign-in detector (checks profiles.metadata.migration_v1_at + Supabase data presence + localStorage data presence). Migration banner render. Batch-import logic (uses Phase D's sync queue). Marks completion. Handles "No, start fresh" path (records decline timestamp; never re-prompts).
Uses Phase D's write paths — no duplicated upload logic.

Total time estimate: 5-7 hours

Combined session beats three separate sessions because the auth state-change pipeline is shared. Single dedicated evening or split across two: (a) Phase C + D scaffolding (Day 1, ~4 hr), (b) Phase F + polish + cross-device testing (Day 2, ~2 hr).

Acceptance criteria (when this bundle is "done")

(a) Anonymous user with 47 local sessions signs in for first time → sees migration prompt with accurate counts → "Yes, import" → all 47 sessions in cloud within 8s.
(b) Same user signs in on a NEW device → no migration prompt (already migrated) → sees hydration overlay → 47 sessions visible after ~3s.
(c) User completes a quiz while signed in → quiz appears in cloud within 30s (next sync flush) or instantly on pagehide.
(d) User goes offline mid-study → sync dot flips yellow → app continues working → reconnect → queued changes flush.
(e) User signs out → local data preserved → can continue anonymously.
(f) User clicks "No, start fresh" once → never re-prompted.
(g) Existing v4.81.2 daily auto-backup mechanism still works (additive, not replacing).
(h) Zero regressions on existing localStorage-driven UX (read latency unchanged).

What's still NOT in this bundle (Phase E + G)

Phase E (server-side API key proxy) — users still BYOK their Anthropic key after this ship. The proxy is a separate concern: it gates HOW MANY requests, not WHO can request.
Phase G (Stripe paywall) — Pro tier features still ungated after this ship. This bundle gives every signed-in user a "free + synced" experience; Pro upgrade is the next layer up.
Both deserve their own dedicated sessions per their issue scopes.
