Three new affordances, one consistent mental model Prev arrow is enabled when there's a question behind you. Next arrow is disabled until you've answered the current one (or if you're already at the last). Dots have numbers now + hover lift + a click handler — click any dot to jump straight there. Keyboard: ← / → mirror the arrows. This matches exam-mode UX so users learn one nav pattern, not two.

At which OSI layer does ICMP operate?
✗ Wrong — correct answer: C
ICMP operates at Layer 3 (Network). It's the protocol used by tools like ping and traceroute to diagnose connectivity. Even though it carries no transport-layer port, it rides directly on top of IP at Layer 3.
✏️ Click a different option to change your answer. Score will update. (Streak stays — you've already seen the answer.)
Editable, but honest Options are re-enabled with their correct/wrong markers still visible. Click a different one to re-pick — log entry updates, score recomputes from the log (e.g. wrong→right bumps score from 1/2 to 2/2). Wrong-bank truth-ups too: changing wrong→right removes from the bank, right→wrong adds. Streak doesn't change on re-pick (you've seen the answer; rewarding it would inflate the streak signal). Yellow banner sets expectations clearly so the user knows score moves but streak doesn't.

Below the transport layer, network is responsible for routing packets across networks; transport (TCP/UDP) sits one layer up and provides reliability + ports. Mnemonic: "Please Do Not Throw Sausage Pizza Away."
✏️ Drag items to rearrange. When you submit again, score updates if you got it right this time.
PBQ revisit — same model, different surface Each PBQ type re-enables its native interactivity: order (drag-to-rearrange), multi-select (toggle picks + Submit again), cli-sim (CLI buttons + diagnosis MCQ re-pickable), topology (device placement re-enabled). Submit fires the existing submitOrder() / submitMultiSelect() / submitTopology() with a "we're updating an existing log entry" branch. Same truth-up rules as MCQ — score recomputes from log, wrong-bank reflects current pick, streak doesn't move.
Implementation summary
Architecture changes (~6 surfaces)
  1. render() — adds an "is this question already in log[]?" check at the top. If yes → render in answered-state with options/PBQ inputs RE-ENABLED (markers preserved). If no → existing pristine path.
  2. pick() + submitMultiSelect() + submitOrder() + submitCliSim() + submitTopology() — each gets an "if log entry exists, update it; else push new" branch. Truth-up score/answered/wrong-bank from the log; streak untouched on re-pick.
  3. NEW jumpToQuestion(idx) + prevQuestion() wired to clickable dots + Prev arrow + ← key. Existing advance() renamed conceptually to "next question" + bound to → key.
  4. _renderQuizProgressDots() — span → button, click handler, numbered labels, hover state.
  5. HTML: 2 new nav-arrow buttons in the quiz top-row. CSS: .qpd-cell hover + clickable styling. .revisit-banner for the yellow strip.
  6. Tests: UAT vm-fixtures for log-update branch (wrong→right truth-up, right→wrong truth-up, streak invariance). Playwright E2E for the full revisit flow (answer Q1, advance to Q2, click dot back to Q1, change answer, verify score updated).
