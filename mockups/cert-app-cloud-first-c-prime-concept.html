☁️ Phase C′ — Cloud-first cert app (revised post-pivot)

Replaces the bundled C+D+F design with a strictly cloud-first model. No localStorage in production for end users. Anonymous trial preserved via session-memory only (lost on tab close). Cert switcher in topbar for multi-cert users. Admin role unlocks /admin dashboard. Phase D and Phase F dissolve into this single ship — no progress sync needed (cloud IS the source), no anon-to-authed migration needed (anonymous users have no persistent local data to migrate).

Five locked-in pivots driving this design

1. No localStorage in production for end users. Cloud-first via Supabase. Eliminates entire class of v4.81-style corruption bugs. Read latency mitigated via in-memory cache (per-session only).

2. Session-memory anonymous trial. Visitor lands → tries sample question → takes diagnostic → sees Pass Plan. State held in JS variables. "Sign up to save this →" CTA at result moment converts the engagement peak. If they don't sign up + close the tab, state is gone — clean.

3. Topbar cert switcher dropdown. Stripe-style account-switcher pattern. Lives between hamburger and date pill. Click → enrolled certs listed → switch navigates to that cert's subdomain (apex cookie keeps session alive across the navigation).

4. Admin role flag + /admin route. Single column on profiles.role TEXT DEFAULT 'user'. Owner's account = 'admin'. Account dropdown shows extra "🔧 Admin Dashboard" link (purple, dashed border to denote admin-only). Routes to certanvil.com/admin with auth gate.

5. Phase F dies, Phase D folds in. No anon-to-authed migration (no anon data to migrate). No separate sync layer (Supabase IS the storage). Net result: simpler architecture overall. Roadmap goes from 4 phases (C, D, E, F, G with D+F bundled) → 3 phases (C′ unified, E, G).

Implementation plan · file map · time
Phase C′ — Cloud-first cert app (single ship, ~9-10 hours)

Combines what was previously C + D + F into one coherent ship now that the local-first complexity is gone.

Files to create / touch (cert app)

app/lib/supabase.js (new) — Supabase client init, copied from landing/lib/supabase.js.
app/auth-state.js (new, ~200 LOC) — auth listener, account-pill render, dropdown render (with admin link if role=admin), sign-out, anonymous-mode detection.
app/cert-switcher.js (new, ~120 LOC) — fetches user's cert_entitlements rows, renders the cert switcher pill + dropdown, handles cross-subdomain navigation.
app/cloud-store.js (new, ~400 LOC) — replaces all localStorage reads/writes with Supabase queries. Implements in-memory caching (per-tab, lost on refresh) for read latency. Provides the same API surface as the existing STORAGE namespace so app.js callers don't need to change.
app/anon-trial.js (new, ~150 LOC) — session-memory state for anonymous users (sample question state, diagnostic state, Pass Plan result). Surfaces signup-gate modal when anonymous user tries to access stateful features.
app.js — replace all localStorage.* calls with cloudStore.*. Guard stateful actions with anonymous check.
index.html — script tags for new modules + topbar markup for cert switcher pill.
styles.css — port account-pill + dropdown CSS, add cert switcher CSS, add signup-gate modal CSS.

Files to create / touch (admin)

landing/admin/index.html (new) — admin dashboard markup. Lives on apex (certanvil.com/admin) so it's cross-cert.
landing/admin/admin.js (new, ~300 LOC) — fetches all-user data, renders stats cards + user table, handles "Manage" action drawer.
landing/admin/admin.css (new, ~250 LOC) — admin-specific styles.
landing/api/admin/users.js (new) — server endpoint that validates admin role then returns user list. RLS policy on profiles already restricts non-admin reads.

Schema additions

profiles.role TEXT DEFAULT 'user' — single column. Owner's account manually set to 'admin' via Supabase dashboard (one-time SQL UPDATE).
RLS policy on profiles: SELECT-all if auth.jwt() -> 'role' = 'admin'; otherwise SELECT-own only.
Same RLS pattern on quiz_history, cert_entitlements, future usage_log.

Acceptance criteria

(a) Anonymous visitor takes diagnostic → sees Pass Plan with prominent signup CTA → if they don't sign up + close tab, state is gone (no localStorage).
(b) Anonymous user clicks any stateful feature (quiz, drill, etc.) → soft signup gate modal with two clear options.
(c) Signed-in user has all features unlocked + data persists in Supabase across devices.
(d) Cert switcher in topbar shows enrolled certs; clicking a non-active cert navigates to that subdomain with session intact.
(e) Admin user (role='admin') sees extra "Admin Dashboard" link in account dropdown.
(f) Non-admin hitting certanvil.com/admin directly → 404 (don't reveal existence).
(g) Admin sees user list, can drill into any user's profile, can manually adjust entitlements / extend pass-guarantee.
(h) NO localStorage writes from production user code (UAT regression-guard via grep).
(i) Existing landing-side localStorage (theme toggle, builder mode, notify-me cache) keeps working — those are landing-page UI prefs, not user data.
