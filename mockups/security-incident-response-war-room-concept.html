Eradication-flavored · destroys evidence
Trap. Power-off destroys all volatile evidence (RAM, running processes, network sessions, decryption keys still resident in memory). Network-isolate first, RAM-dump second, only THEN power down if needed. This is the #1 SY0-701 IR trap.

Containment · scope expansion · MISSED
Required — you missed this. Containment isn't just "stop the obvious thing" — it's "find the unknown spread." This is what surfaced HR-WS-04 (the second infected host now on the right-panel feed).
!
The power-off trap
SY0-701 Domain 4 · the most common containment-phase exam mistake
"Powering off the infected host stops the encryption" sounds right — and it does stop encryption — but it destroys all volatile evidence in the process. RAM holds the running ransomware process state, the decryption key (sometimes), injected DLLs, the C2 session, and recently-accessed file tables. Network-isolate achieves the same containment goal (no more spread, no more C2) while preserving everything forensics needs to attribute the attack and possibly recover keys. Memorize: isolate ≠ power off.

STATE 7 · Lessons tab — 6 PICERL phase cheatsheets

Goal statement, 3-5 canonical actions, then the trap-callouts (the 1-2 mistakes SY0-701 LOVES to test on this phase). Cards are self-contained reference material — print-friendly + searchable.
P1
Preparation
"You can't pause an incident to write a playbook." Build readiness BEFORE day-zero.
  • IR plan documented + reviewed annually
  • IR team roles assigned (commander, scribe, comms, legal, forensics)
  • Tooling staged (EDR, SIEM, NetFlow, write-blockers, RAM-capture)
  • Tabletop exercises quarterly with realistic scenarios
  • Backups + comms tree tested + out-of-band channel ready
⚠ Common traps
"Buy more tools." Tooling without playbook + practice doesn't help. Process > products.
P2
Identification
"Is this an incident? What kind? How bad?" Confirm and classify.
  • Confirm via correlation across sources (EDR + FW + user report)
  • Classify incident type (malware / data breach / DoS / etc.)
  • Assign severity per the IR plan's matrix (impact × urgency)
  • Open ticket + start the scribe log + start incident timer
  • Stay broad — collect IOCs even if you're not sure they're related yet
⚠ Common traps
Jumping to containment too early. Without classification, you over-isolate or under-isolate. Classify first.
P3
Containment
"Stop the spread. Preserve evidence. Don't make eradication harder."
  • Network-isolate infected hosts via EDR (NOT power off)
  • Block C2 at perimeter / DNS sinkhole
  • Identity-contain — disable compromised accounts + force re-auth
  • RAM dump before any state-changing action (RFC 3227 order of volatility)
  • Scope-expand — sweep for lateral movement (other hosts hit?)
⚠ Common traps · the famous ones
Power-off vs isolate. Power-off destroys RAM, swap, network state, decryption keys. Isolate keeps the host alive for forensics. Never power off first.
Wipe-and-reimage too soon. That's eradication. Wiping pre-evidence-capture loses attribution + IOCs.
P4
Eradication
"Remove the malware AND the way it got in. If you only remove the malware, it'll come back."
  • Wipe-and-reimage (preferred over disinfection — trust nothing)
  • Patch the exploited vulnerability (not just the host you found)
  • Rotate credentials for any account that touched the affected host
  • Revoke certs if private keys were on the box
  • Update IOC blocklists permanently — promote from temporary block
⚠ Common traps
"Just run AV again." Disinfection isn't trust-restoring. Reimage from clean baseline.
Patching only the affected host. Same vuln exists on every other host. Patch fleet-wide.
P5
Recovery
"Get back to business safely. Watch for recurrence." Restore + monitor + validate.
  • Restore from clean backup verified pre-incident timestamp
  • Validate integrity — hash-check restored files vs known-good
  • Stage re-introduction — bring services back gradually + monitored
  • Heightened monitoring for ~30d post-recovery (recurrence risk)
  • User communication — what was affected, what to watch for
⚠ Common traps
Restore before eradication. Restoring onto an unpatched host re-infects. Sequence matters.
Restoring from compromised backup. Pre-incident timestamps only. Verify backup hashes.
P6
Lessons Learned
"How do we make sure this never happens again — or at least costs less next time?"
  • Post-incident review within 2 weeks while details fresh
  • Document timeline — every action + when + by whom
  • Identify gaps in detection, response, comms, tooling
  • Update playbook with new attack-pattern + better response
  • Share intel with peer orgs / ISAC if appropriate
⚠ Common traps
Skipping it. "We're too busy" = next incident is the same incident. Always close the loop.
Blame culture. Blameless postmortems surface real gaps. Punitive ones surface lies.
