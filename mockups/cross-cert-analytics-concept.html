🧱 Design decisions to sign off (or push back)

Decision A — Page, not modal. Move from the current launcher modal to a real page at certanvil.com/analytics/cross-cert. Modal stays as the entry chip on the dashboard but routes here. Rationale: real analytics needs scrolling, comparison side-by-side, and hover/click into details. Modals fight all three.
Decision B — Lives on landing (certanvil.com), not cert app subdomain. Cross-cert is umbrella-by-definition; landing is the cert-agnostic surface. Cert app is per-cert. Going to networkplus.certanvil.com for cross-cert analytics doesn't make sense.
Decision C — Skill overlap is hand-authored, not computed. A CROSS_CERT_OVERLAP table lives in landing/lib/. ~15 cert-pair entries (C(6,2)=15 pairs across 6 certs). 1 Saturday's work to populate, gives high signal. Computed similarity from topic-name strings gets it wrong — "Wireless" vs "Wireless security" vs "802.11" are the same concept but different strings.
Decision D — "Readiness" reuses cert-app's existing 420–870 scaled score (getReadinessScore() at app.js:3066). Pulled from each cert's quiz_history via Supabase RPC. Single source of truth across cert app + landing.
Decision E — Recommendations are a deterministic ranker, not LLM-generated. Inputs: per-cert readiness, days to exam, weakest domain, exam-weight × accuracy gap, last-practiced staleness. Output: ranked list of next actions. Same prescriptive philosophy as Subnet Trainer dashboard. LLM gets it wrong on math; deterministic doesn't.
Decision F — Passed certs are visually unmissable. Three layered treatments so there's zero ambiguity: (1) Passed rows are in their own group at the top of the side-by-side panel with a green divider header (✓ Passed (N)); (2) each passed row gets a left-border accent + green-tinted background + a corner badge (🏆 PASSED) anchored top-right; (3) the readiness "score" reads as final scaled score (e.g. 767/900 in green) rather than the in-progress readiness number. Pulled from profile.metadata.cert_results.<cert>.status === 'passed' — single source of truth across cert tiles, /account exam-results, and this dashboard.

What's NOT in v1 (defer to v2 if there's appetite)

  • Predicted pass date — "if you keep this pace, you'll be exam-ready on May 28." Needs more historical pace data than current users have.
  • Heatmap of study time per cert per day — interesting but not action-driving; classic "descriptive without action" trap (per Analytics Prescriptive principle in CLAUDE.md).
  • Cross-cert exemplar comparison — "you got SQL injection wrong on Sec+; here's the related pattern in AWS SAA." High value but requires per-question analysis pipeline; v3+ work.
  • Team/org view — collaborative cross-cert dashboards. Saas-gated to post-launch when there are real teams.

Implementation phases (post-mockup signoff)

  • Phase A — Skeleton page + Panel 1 (pass-readiness side-by-side). Reuses existing readiness score. ~1 day mockup-to-shipped.
  • Phase B — Panel 2 skill overlap. Author the CROSS_CERT_OVERLAP table (Saturday work). ~1.5 days.
  • Phase C — Panel 3 next-up recommender. Deterministic ranker w/ test fixtures. ~1.5 days.
  • Phase D — Replace existing modal: launcher chip on dashboard now routes to the new page; modal becomes a light teaser preview. ~half day.

Total: ~4-5 feature-days across 2 sprint weeks. Slots into Mon/Wed/Fri feature lanes.

Open questions for sign-off

  • Is "Cross-cert analytics" the right name, or does Cert overview / Study dashboard / something else feel cleaner?
  • Does the "hours saved" estimate on skill overlap feel real, or does it need to be backed by something concrete (vs gut-feel)?
  • For empty state — should the user be auto-enrolled in N+ (most-popular cert) or pick from the cert grid?
  • Power-user state shows 4+ active certs sorted by exam date. Acceptable, or should we group by status (active / soon / passed) with sub-sort by date?
