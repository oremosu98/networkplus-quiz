Design Decisions · Implementation Plan
Why magic-link first, password later (or never) Magic links solve the password reset spiral. ~30% of "I can't sign in" support tickets across SaaS are forgot-password related. With magic links, every sign-in IS a password reset — no separate flow. Slightly slower per-session (email round-trip) but the UX dependency tree is minimal. Stripe, Notion, Linear all started magic-link-only and only added passwords years later. We can add passwords if customer demand surfaces; we don't need them at launch.
Why same flow for sign-up + sign-in "Don't have an account?" links create branching cognitive load. Magic links remove that — Supabase auto-creates an account if the email is new, sends the magic link either way. The auth-sub copy says "New here? Same form — your account gets created automatically" so users aren't confused.
Cross-subdomain session via apex cookie Supabase's auth client sets the session cookie on .certanvil.com (apex with leading dot). All subdomains inherit the cookie. Sign in once at certanvil.com → networkplus.certanvil.com sees the same session → cert app shows logged-in state automatically. No re-authentication, no token-passing dance.
Anonymous → authed migration Three trigger points where we offer the migration banner: (a) when an anonymous user clicks "Sign in" with localStorage progress detected, (b) when an authed user signs in on a fresh device that DOES have anonymous localStorage progress (ask "Use this device's local progress?"), (c) post-signup if the user took the diagnostic anonymously first. Critical UX: never auto-merge without consent. Always show what's getting imported (47 sessions / streak / readiness) so the user decides knowingly.
What's NOT in this mockup yet Account settings page (deferred — placeholder route). Cross-cert analytics dashboard (Phase E or later). Email change flow (rare, ship when needed). Two-factor auth (defer — magic links already require email access, which is most of 2FA's value). Email preferences / unsubscribe (when we add marketing emails). Delete account flow (legally required eventually but not for launch — can be a manual "email us" for now).
Implementation scope (when you sign off) Magic-link flow: ~3 hours. Account dropdown: ~1 hour. Cross-subdomain cookie + auth state propagation: ~1 hour. Migration banner + flow: ~2 hours (Phase D territory but tile-aware). Total Phase B: ~6-8 hours of focused work, splittable across 2-3 sessions.
