🧱 Design decisions to sign off (or push back)

Decision A — Chip lives in the topbar, not the sidebar or the quiz screen. Topbar is always visible across all cert-app pages (Home, Practice, Drills, Topology, etc.) — quota awareness should be persistent, not page-specific. Sidebar feels too far from the action; in-quiz is too contextual.
Decision B — Reset at midnight UTC, not local time. Single global reset moment makes server-side enforcement trivial (just a current_date in Postgres). Local-time reset would require timezone metadata per user + edge cases on travel/DST. UTC is "honest" and the chip surfaces "Resets in 4h 23m" so users see a real countdown anyway.
Decision C — Quota-exceeded is an inline replacement, not a modal. When the user clicks "Start quiz" at 20/20, we replace the quiz body with the exceeded card. Modals are interruptive ("you can't do this thing") — replacement explains why ("here's why you can't do this thing") and sells Pro contextually. Less feels-like-a-trap.
Decision D — One-time migration banner for existing BYOK users only. Users with no STORAGE.KEY see nothing new (they just enjoy server-handled AI). Users with a stored key see the banner once, can dismiss or delete the key. Stored key is silently ignored regardless — banner is purely a privacy/awareness gesture.
Decision E — Quota counts fetchQuestions calls only, NOT validation/coaching/teacher. Validation second-pass + explain-further + coach-this-topic are infrastructure costs we eat. Counting them would make "20 questions/day" feel like "5 questions/day in practice." The user-facing count matches the user-facing number.

What's NOT in this mockup (deferred to Phase E.5/E.6)

  • The quota dashboard view (history of daily usage, which days you hit the cap, per-cert breakdown). Optional polish — defer until users actually ask.
  • Pro fair-use throttling UI — what does the chip look like for a Pro user who hit the soft cap of ~200/day and got throttled to 50/hr? Probably another red-tinted state similar to the Free exceeded view but with different copy. Will design when that scenario actually happens (Phase E.6, post-launch).
  • Anonymous-user quota. Anonymous users can't be quota-tracked server-side because there's no user_id. Decision needed: do anonymous users get any quiz access at all, or is "sign up free, get 20/day" the only path? Current default: signed-in only, anon sees the homepage CTA to sign up. Worth confirming.
