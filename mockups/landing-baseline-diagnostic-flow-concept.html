01

Why move the diagnostic to the landing page

The current funnel has a hidden conversion-killer that we never built around. This fixes it.

Current state · the problem

A visitor lands at certanvil.com wanting to "try the diagnostic." Today they have to:

  1. Click into a cert-specific page
  2. Get pulled INTO the cert app subdomain (networkplus.certanvil.com)
  3. Hit the cert app shell (which is now 200 KB+ even after Phase 11 extractions)
  4. Discover the diagnostic CTA buried among 15 other features
  5. Need to set up an API key OR sign in to access AI features

Drop-off cliff. Every step asks for more commitment before delivering any value.

Proposed · the fix

A visitor lands at certanvil.com and sees:

  1. Hero CTA: "Take the Baseline Diagnostic" — primary action above the fold
  2. One click → cert picker (currently Network+; admin sees Security+ too)
  3. One click → diagnostic starts on a clean, dedicated page
  4. 30 min later → score + 5 outcome paths including subscribe to Pro

No account required. No cert app shell to download. Conversion-ready surface. Honours the PLG plan.

05

Anti-abuse · anonymous AI quotas

Anonymous AI access is a gift to abuse. Without guardrails the diagnostic endpoint becomes a free AI proxy. Here's how we lock it down without hurting real users.

Server-side

1 diagnostic / IP / 24 hours · anonymous

Tracks IP address (hashed, not stored long-term) in a Redis-lite Supabase table. A second diagnostic attempt from the same IP within 24h gets a 429 with "Cooldown · sign in for unlimited retakes." Real students cooperate; abusers walk away.

Server-side

Turnstile / hCaptcha on quiz start

Cloudflare Turnstile (preferred — invisible, ~free) or hCaptcha invisible challenge before the first AI call fires. Validates the user is a real browser, not a script hitting /api/ai/generate in a loop. Doesn't interrupt real users.

Server-side

Anonymous quota cap · 25 AI calls / diagnostic

Diagnostic needs ~25 calls (20 Q + buffer for filtering). Server-side hard cap = 25. After that the /api/ai/generate route rejects with 429. Hardcoded — can't be bypassed by retrying.

Cost

Estimated $ per anonymous diagnostic

~25 Haiku calls + ~5 Sonnet validations = roughly $0.10-0.25 per diagnostic. Affordable as a marketing cost until 1000+/day. At that scale, lower the anonymous cap or require signin earlier.

UX

Show progress · "23 of 25 AI calls remaining"

Transparent quota meter visible on intake screen. Frames it as "you have everything you need" not "we're rationing." Builds trust.

Worst case

If we get hammered · graceful degradation

If anonymous quota for the day is exhausted (e.g., we hit a 10x traffic spike), the diagnostic falls back to a pre-baked question pool (subset of the curated exemplar bank) instead of live AI generation. Lower quality but still useful — and we explicitly tell users "high traffic right now, results may use pre-generated questions."

09

Edge cases · things that could go wrong

What happens when a user does the unexpected. Each one needs explicit handling — or an explicit "we accept this is broken in v1."

Resume

User starts, closes tab mid-question

Tab close = sessionStorage cleared. On next visit, the score-snapshot in localStorage (if they got far enough) shows but the in-progress quiz is gone. Toast: "You started a diagnostic 2 hours ago but didn't finish. Want to start fresh?" → restart from Q1.

Resume

User reloads page mid-quiz

Page reload preserves sessionStorage. The quiz resumes at the current question. We show a "Welcome back" indicator briefly so they know it picked up.

Multiple certs

Anonymous user takes Network+ diagnostic, then wants to take Security+

Multi-cert anonymous support stores all-diagnostics-taken in localStorage as an array. Picker shows ✓ on certs they've already done with score; they can retake or proceed. No state collision.

Cooldown

User retakes within 24h to try to score better

Anonymous 24h cooldown shows "Come back in 4 hours · or sign up for unlimited retakes." Signed-in users have no cooldown.

Abandoned signup

User clicks "Unlock Pro" but doesn't complete checkout

If they bail at the Stripe page, they're left as a signed-in free user with their diagnostic results in their profile. Win-win: we still captured the lead, they still have access to their data, and they can come back any time to upgrade.

AI down

Anthropic API is down or rate-limited

Server returns 503 → diagnostic UI shows "We're experiencing high traffic right now. Try again in a few minutes." User retains their progress. Bonus: pre-baked fallback question pool kicks in (see §05).

Mobile

30-min quiz on a phone

Wake Lock acquired (Phase 10 v4.99.49 pattern) so phone screen stays on. Visibility API pauses timer if they switch apps. Same exam-mode discipline as the cert-app version.

Already a user

Signed-in user opens /diagnostic from landing

Treat them as taking the in-app diagnostic — but on the landing surface. On submit, results auto-merge into their profile (skip the migration step). They see "Continue in cert app →" as the primary path instead of "Subscribe."

Bot/spam

Headless scraping the AI proxy

Turnstile invisible challenge + IP cooldown + anonymous quota cap. Three independent guards. If bot traffic still gets through, we add explicit hCaptcha visible challenge — slight friction to real users but kills bots.

No JS

User has JS disabled

Diagnostic can't run without JS — show a graceful "this page requires JavaScript" message with a link to enable it in their browser. Crawlers see meta-description fallback for SEO.

Multi-tab

User opens diagnostic in two browser tabs simultaneously

sessionStorage is tab-scoped, so each tab starts its own quiz. Both can submit independently. Anti-abuse: anonymous quota is server-side, so they can't get 50 questions instead of 25.

SEO

Search engines indexing the diagnostic

Each route has its own meta-description + OG image + structured data. /diagnostic = "Take CertAnvil's free Network+ diagnostic in 30 minutes." Results pages (/diagnostic/r/{token}) are noindex to prevent leaked scores becoming search results.

You asked what you might have missed. Here's what I'd flag for explicit decisions before implementation.

Should we ask for email at the start of the quiz (so we capture leads even if they don't reach the results page)? Social-share quote-cards

Cohort comparison

"Your 680 puts you in the top X% of recent test-takers" — only meaningful once we have 50+ diagnostic completions. Defer to v1.2. Adds urgency: "I'm above average · let's lock it in."

Tech debt prevention

Reuse v4.81.0 question UI components

The cert-app already has good Q-card markup (the cert app's diagnostic page). We extract just the rendering logic into a small standalone component — same look + feel, but no dependency on the cert-app shell. Future diagnostic UI tweaks made once, applied to both surfaces.

Add

Auto-resume modal on re-visit

User came back to certanvil.com after taking the diagnostic. Show a soft modal: "Welcome back · last week you scored 680 on Network+ · want to keep building from there?" → CTA to sign up or retake. Re-engages bookmark traffic.

Decision needed

What counts as the canonical "first diagnostic"?

If user takes landing diagnostic anonymously, signs up, then takes the in-cert diagnostic 2 weeks later — which one is their "baseline"? Recommendation: the SIGNED-IN version becomes the canonical baseline. Anonymous data is a one-off snapshot.

Compliance

Anonymous-AI privacy disclosure

Server stores IP-hash (for rate limiting) + diagnostic content (for the shareable link). Both should be mentioned in Privacy Policy. Action: add to the Phase-6b privacy doc in a v1.1 amendment.

Performance

Pre-warm the AI question cache

Background cron generates a pool of N+ baseline-diagnostic questions every hour. When a user starts, the first 5 questions come from the warm pool (instant render), then live generation kicks in for the remaining 15. UX = "the diagnostic feels snappy."

Add

Analytics events for conversion tracking

Fire events on each step: diagnostic_started · cert_picked · intake_completed · quiz_q1_answered · quiz_completed · results_viewed · cta_clicked (with path A-E). Lets you measure the funnel + see which step has the biggest drop-off. Recommendation: use the existing web_vitals collector + extend with custom events.
