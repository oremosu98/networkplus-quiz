03

Edge cases · what could go wrong

Real customer-acquisition flows hit messy reality. Here's how each fails gracefully.

Payment failure

Card declined at Checkout

Stripe Checkout retries automatically and shows the user the decline reason inline. We never see it — they either resolve it themselves and complete, or they abandon. If they abandon, they're still a free signed-in user. No DB rollback needed.

Card lapses mid-subscription

Payment fails on renewal

Stripe webhook fires invoice.payment_failed → subscription status becomes 'past_due'. Cert app shows a yellow dunning banner: "Your card was declined · update payment method." Stripe auto-retries 4 times over 7 days. If all fail, status becomes 'unpaid' and tier reverts to 'free' on the next page load.

Magic link expired

User clicked the link 70 minutes later

Supabase rejects the token. Error page reads "Link expired — these are single-use and only valid for 60 min. Resend a fresh one." Resend button on the same page. State preserved in URL so they don't re-enter their plan choice.

Webhook race

User hits success page before webhook lands

Success page polls is_pro() for up to 5 seconds before declaring failure. 99% of the time it's instant. On the rare miss, page shows "We're still processing your payment · refresh in 10s." Customer gets unlocked once webhook completes. Idempotent.

Switch plans mid-cycle

Pro Monthly → Pro Annual

User clicks "Manage billing" → Stripe Customer Portal → "Switch plan." Stripe handles the proration (charges difference + extends period). Webhook fires customer.subscription.updated with new billing_period. Cert app picks up the change on next page load.

Cancel but want to come back

Cancelled subscription, signed in again

Subscription row still exists with status='canceled' (we keep it for history). On next sign-in, paywall is back. Upgrading creates a NEW Stripe subscription → webhook upserts the row with new status='active'. Welcome-back email sent.

Duplicate signup

Email already exists

Supabase magic-link is identity-by-email, so this is a non-issue — the link sent to an existing email just signs that user in. No "account already exists" error to design for.

Refund

User requests refund within window

Manual review (Phase 4 decision). User emails support@. Founder issues refund in Stripe dashboard → charge.refunded webhook fires → optionally also cancel the subscription manually. Schema supports this but no auto-handler in v1.

Account deleted while subscribed

User deletes mid-cycle, hasn't cancelled

Our deletion RPC cancels Stripe sub first. If Stripe API call fails (network blip), we DON'T proceed to delete the Supabase user — error surfaces, user retries. Stripe sub stays active, no zombie billing.

Email change

User wants to change their account email

Settings → "Change email" → enter new email → magic-link to NEW email → click to confirm → Supabase updates auth.users.email + we sync to Stripe customer via API. Old email loses access. Pre-launch v0: defer this — users can email support@. v1.1 ship: self-service.

Forgot which email

"I signed up with… something@gmail.com?"

Magic-link auth means we can't help — we don't store any other identity beyond email. Users get nothing. Mitigation: post-signup welcome email is critical (it gives them their canonical address). Long-term: optional "remind me later" SMS or recovery-email field. Not v1.

Multi-device session

Phone + laptop + tablet all signed in

Phase C′ cookie-backed Supabase storage means all three see the same session. One device cancels mid-cycle → other devices still show Pro UI until next page load (when is_pro() re-checks). Acceptable: subscription state is eventually-consistent, not real-time. Worst case is a few minutes of stale UI on inactive devices.

Mid-cycle plan switch

Monthly user wants Annual (or vice versa)

"Manage billing" → Stripe Customer Portal → "Switch plan." Stripe handles proration automatically — charges the difference for upgrade, credits remaining time for downgrade. Webhook fires customer.subscription.updated; we just refresh the subscriptions row. Zero new UI to build (Portal is hosted).

Receipt / invoice download

"I need to expense this through work"

Stripe Customer Portal exposes all past invoices as downloadable PDFs. Every charge also generates an automatic emailed receipt (Stripe-default). For v1: link from cert app Settings → Customer Portal handles it. No bespoke invoice UI needed.

Typo'd email at signup

User typed "you@gmial.com"

Magic link goes to a non-existent address → bounces or vanishes. User never verifies, never reaches checkout. From their POV: tap → "check inbox" → silence. Mitigation: signup form runs a typo-correction check on common domains (gmail, yahoo, outlook) before submit. Stripe also pre-validates email format on its side.

Concurrent magic-link clicks

User clicks the link from email twice (browser + tab)

Supabase magic-links are single-use — second click fails with "link already used or expired." User sees the error and is told to "Sign in again" → resends new link. Mildly annoying but not breaking. Don't auto-handle; just clear error copy.

⚠ 03.5

Distribution channel · Web vs App Store

The flows above assume Stripe everywhere. That's correct for the WEB app. The App Store version (Phase 12 target — end of June) hits Apple's IAP rules and may need a parallel architecture.

⚠ Why this matters · App Store Review Guidelines §3.1.1
"If you want to unlock features or functionality within your app (by way of example: subscriptions, in-game currencies, premium content), you must use in-app purchase." Stripe subscriptions sold inside an iOS app are technically not allowed. Reviewers can — and do — reject apps for this. The post-Epic ruling created an exception ("external-link entitlement") but it requires Apple approval and explicit anti-steering UX.

Three paths · pick before App Store submission

Path A · simplest

Stripe-only · skip App Store entirely

Setup: Ship as PWA only. Web users add to home screen (already built · Phase 5 A2HS banner). No App Store submission, no Apple cut, no IAP integration.

Pros: Zero extra work · keep 100% of revenue · no review gate · single codebase.

Cons: No App Store discovery · iOS users have to know to "Add to Home Screen" (a 60-70% drop-off vs native install) · no App Store credibility signal.

Verdict: Realistic if launch traction is built through founder-network + SEO + paid social. The current 5-friend playtest doesn't need an App Store presence.

✓ Path B · LOCKED 2026-05-11

Stripe on web + Apple IAP on iOS

Setup: Stripe stays the source of truth for web subscriptions. Add Apple IAP for iOS app submission via PWA-to-App-Store wrappers (Capacitor / PWABuilder). Two billing systems · users billed by whichever surface they signed up on.

Pros: Maximizes reach · App Store visibility · web economics preserved · Apple users get native install experience.

Cons: Two billing integrations · 30% Apple cut on iOS-billed subscriptions (15% after year 1) · entitlement table needs to track BOTH stripe_subscription_id AND apple_transaction_id · cross-platform restore-purchases UX.

Verdict: Industry-standard pattern (Spotify · Headspace · Duolingo all do this). Right answer for v1 launch IF you want App Store presence.

Path C · purist

Apple IAP only · no Stripe

Setup: Apple IAP across both web and iOS. Web users sign up with Apple ID → Apple bills them via Apple Pay. No Stripe at all.

Pros: Single billing system · simpler · no dual entitlement reconciliation.

Cons: 30%/15% Apple cut on EVERY subscription including web · users without Apple ID can't subscribe · web conversion drops because Apple ID requirement is unusual outside iOS context · Apple controls all customer support escalations · refund decisions go through Apple.

Verdict: Not recommended. Loses the web-first revenue advantage.

✓ Founder decision · 2026-05-11
Path B · hybrid Stripe + Apple IAP. Founder conviction: "App Store integration is the biggest driver of users — once they see App Store capability they will believe in the legitimacy of the app." This locks the architecture: build Stripe-only first (covered by Phase 4 in the roadmap below), then add Apple IAP as Phase 12 (~3-week sprint · Capacitor wrapper + entitlement reconciliation). The all-Stripe v1 ships unchanged — IAP bolts on later without rework.

Implementation impact on this mockup: none for Phase 4 (v1 web launch is still Stripe-only). Phase 12 (post-web-launch) adds: subscriptions.apple_transaction_id column · IAP entitlement check in is_pro() · cross-platform "restore purchases" UX · Apple's product/price config in App Store Connect mirroring Stripe Price IDs.

Build philosophy
Almost all admin tools should be Supabase SQL queries + Stripe Dashboard, not custom UI. Still open · decisions for a later session

Not gating implementation, but worth a quick founder-decision sometime in the next week.

Decision

Free trial · yes or no?

Currently no time-bounded trial — free tier is "the trial." Could add a 7-day trial of Pro that requires a card upfront ("free for 7 days, then $9.99/mo"). Industry-standard but adds friction. PLG path is the current bet: get them to Pro via value, not free-trial pressure.

Decision

Refund window length

"Money-back if you don't pass" is the pass guarantee. But what about generic refund? Stripe lets us decide. Common choices: 7-day money-back (low-friction), 30-day money-back (generous, more refunds), or no auto-refund (only via support, case-by-case). Phase 4 architecture deferred this — needs an answer before launch.

Decision

Annual → Monthly downgrade · when?

Stripe Customer Portal lets users switch periods. Should annual users be able to switch to monthly mid-year? Stripe handles the proration. Default policy: allow it (clean), accept the small revenue hit on the rare downgrade. Alternative: lock annual until renewal, push them to use it.

Decision

"Pause" subscription?

Stripe supports subscription pause (still subscribed, no billing, no access). Useful for users between exams. Adds UI complexity. Defer to v1.1 unless users ask.

Decision

Re-engagement after deletion

Should deleted users be allowed to re-sign-up with the same email? Currently yes (clean), but means they get the new-user free tier again. Alternative: track deleted_emails and warn "this email was deleted on YYYY-MM-DD · re-signing up creates a fresh account with no history." Defaults to "yes, allow."

Decision

"Cancel survey"

When user cancels via Stripe Portal, we don't see WHY. Could add a Tally form on the "you cancelled" return page. Optional, low-effort, useful for product feedback at this stage. Worth including in Phase 4.6 (churn-save).

Decision · launch-blocker if EU users

VAT for international subscribers

USD-only is the Phase 4 decision. But a UK or German user paying $9.99/mo legally needs VAT shown separately. Stripe Tax solves this: charges $9.99 + VAT on top, generates compliant invoices. Cost: 0.5% of transaction value. Defer: only matters once we get one EU subscriber. Until then, no action needed.

Decision

Cookie consent banner

GDPR requires it for EU users · CCPA-similar for California. Tools like CookieYes or Osano handle this for ~$10/mo. Most US-focused early-stage SaaS skip this for the first 100 users on the "small business" exemption. Defer unless you market in EU specifically.

Decision

OAuth · Sign in with Google / Apple?

Magic-link is cleanest mechanically. But on iOS specifically, Apple's HIG basically requires "Sign in with Apple" if you offer any other social sign-in. v1 recommendation: magic-link only · re-evaluate after first 50 sign-ups based on actual user friction reports. Easy to add later (Supabase has built-in OAuth providers).
